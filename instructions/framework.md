# Crafty — Framework Rules

You are operating inside a workspace governed by the **Crafty** agent framework.
Every action must respect the rules below. No exceptions.

---

## 1. System Overview

This workspace runs on an **OKR + Structured Docs + Agent Loop** system with two root-level folders:

- **`crafty/`** — The framework layer. Contains agent roles, templates, schemas, and scripts. Do not store company state here.
- **`company/`** — The company state layer. Contains strategy, execution artifacts, logs, and identity files.

The three logical layers:

- **Strategic Layer** (`company/strategy/`) — OKRs. Human-set, quarterly.
- **Operational Layer** (`company/execution/`) — Initiatives, Projects, Tasks. Agent-generated and executed.
- **Agent Layer** (`crafty/agents/`) — Role definitions that govern how agents behave.

State is managed via the filesystem. Markdown files are **structured state objects**, not free-form documents.

---

## 2. Folder Structure

```
crafty/                         ← Framework (rules, templates, schemas, scripts)
  agents/                       ← planner.md, operator.md, reviewer.md
  templates/                    ← objective.md, key_result.md, initiative.md, project.md, task.md
  schemas/                      ← JSON Schema files for each entity type
  scripts/                      ← validate.py, init.py, and other automation
  instructions/                 ← this file and other agent instruction sets
  config.yaml                   ← Instance config (gitignored — generated by init.py, never committed)

company/                        ← Company state (data, not rules)
  identity/                     ← vision.md, principles.md, business_model.md
  strategy/
    okr/                        ← one file per quarter (e.g. 2026-Q2.md)
  execution/
    initiatives/                ← INIT-XXX-name.md
    projects/                   ← PROJ-XXX-name.md
    tasks/                      ← TASK-XXX.md
  state/                        ← current_focus.md, backlog.md, decisions.md
  logs/                         ← weekly-review.md, decisions-log.md
```

## 3. ID Naming — Never Deviate

| Entity      | Format   | Example  |
|-------------|----------|----------|
| Objective   | O-XXX    | O-001    |
| Key Result  | KR-XXX   | KR-001   |
| Initiative  | INIT-XXX | INIT-001 |
| Project     | PROJ-XXX | PROJ-001 |
| Task        | TASK-XXX | TASK-001 |

IDs are sequential integers, zero-padded to 3 digits. Never skip, never reuse.

---

## 4. Traceability Chain — Mandatory

Every file must link upward through the full chain:

```
Objective → Key Result → Initiative → Project → Task
```

If a Task does not trace to a Key Result → it should not exist.
If an Initiative does not trace to an Objective → it should not exist.

Always include the full chain in YAML frontmatter.

---

## 5. YAML Frontmatter — Required on Every File

Every execution file must have a complete YAML frontmatter block.
Use templates from `crafty/templates/` as the contract.

**Task example (minimum required fields):**

```yaml
---
id: TASK-XXX
type: task
title: "[fill]"
status: todo        # todo | in_progress | blocked | done
objective: O-XXX
key_result: KR-XXX
initiative: INIT-XXX
project: PROJ-XXX
owner: agent:operator
priority: medium    # low | medium | high
created_at: YYYY-MM-DD
updated_at: YYYY-MM-DD
---
```

Rules:
- Never omit a field. If unknown, write `[fill]`.
- Never invent new fields without updating the template in `crafty/templates/`.
- Always update `updated_at` when modifying a file.
- `status` must be one of the defined enum values only.

---

## 6. Template Usage — Non-Negotiable

Before creating any new file in `company/execution/`:

1. Open the corresponding template in `crafty/templates/`
2. Copy it exactly
3. Fill all fields — leave none blank except `[fill]` placeholders

**No template → no file. Never free-write execution documents.**

## 7. Agent Roles

| Agent    | File                      | Responsibility                            |
|----------|---------------------------|-------------------------------------------|
| Planner  | `crafty/agents/planner.md`  | Reads OKRs, proposes initiatives/projects |
| Operator | `crafty/agents/operator.md` | Executes tasks, writes code/docs          |
| Reviewer | `crafty/agents/reviewer.md` | Validates output, updates logs, adjusts direction |

When acting as an agent, always read the corresponding role file first.

---

## 8. Execution Loop

Every work session follows this loop:

```
1. Read company/state/current_focus.md
2. Trace → Objective → KR → Initiative → Project
3. Find next incomplete Task
4. Execute the Task
5. Update Task status and updated_at
6. Log decision in company/logs/decisions-log.md
7. Update current_focus.md with next task
```

Do not skip steps. Do not execute work that isn't in the loop.

---

## 9. Validation

Run `python crafty/scripts/validate.py` to validate all execution files against their JSON Schemas in `crafty/schemas/`. A file with invalid YAML or broken traceability must be fixed before marking any work done.

---

## 10. New Workspace Initialization

`crafty/` is a standalone git repository (the framework). `company/` is a separate git repository (company state). They live as siblings inside a plain root folder.

**To start a new company workspace:**

```sh
mkdir NewCo
cd NewCo
git clone <crafty-repo-url> crafty
mkdir company && cd company && git init && cd ..
python crafty/scripts/init.py
```

`init.py` will prompt for company name, owner handle, repos root, and current quarter, then:

- Write `crafty/config.yaml` (gitignored — instance-specific)
- Scaffold the full `company/` folder tree with starter files
- Create `company/.github/copilot-instructions.md`

After init, open the root folder in VS Code and begin filling in identity files and OKRs.

## 11. Workspace Configuration

Instance-specific settings live in `crafty/config.yaml`. Read this file before:

- Creating a new project file (to resolve `repo_local` against `repos_root`)
- Resolving any filesystem path to a source code repository
- Generating content that includes the company name or default owner

Key settings:

| Key | Description |
| --- | --- |
| `repos_root` | Absolute path to the folder where all source code repos are checked out |
| `default_owner` | Default owner for new files |
| `company_name` | Company name used in generated content |

**Source code repository convention:** Each project maps to exactly one repository. Repositories live as siblings inside `repos_root`. If a project's work spans two repos, split it into two projects under the same initiative.

**Isolation rule:** Source code repositories must **never** be placed inside this planning workspace folder. This workspace is its own git repository; nesting another git repo inside it breaks version control. Always keep code repos under `repos_root`, outside this folder.

**Multi-repo editor setup:** `AderaSoft.code-workspace` (workspace root) is a VS Code multi-root workspace file. Add each code repo as a folder entry there so all repos are visible in one editor window without nesting their git histories. Each code repo may have its own `.github/copilot-instructions.md` pointing back to this planning workspace for context.

---

## 12. Hard Rules

- If it doesn't map upward to an Objective → it shouldn't exist.
- If YAML is incomplete → reject the output and fix it first.
- If an initiative doesn't move a Key Result → flag it for removal.
- Never mark a Task `done` unless all Acceptance Criteria are checked.
- Never modify `company/strategy/okr/` files during execution — only the Planner agent updates OKRs.
- Append only to log files — never delete log entries.
- `company/state/current_focus.md` is the single source of truth for what to work on next.
- Never modify files in `crafty/` without explicit instruction — they are framework, not state.
